Reactive Programming with Actors

Johannes Pelto-Piri

Reactive Programming

  • Applications should
    • React to events
    • React to load
    • React to failures
    • React to users

Why?

  • Do you need to build reactive applications?
  • Do you need to build fault tolerant applications that can repair themself?
  • Do you need to build distributed applications?

A actor based system will help you with this

This is what you get:

  • Actor systems contains a hierachy of actors that works concurrently and process messages independedly.
  • Actors can supervise other actors and repair, restart or terminate them when they fail.
  • Actor systems provides you with location transperancy, so your system can span multiple machines

Concepts

Actors

  • A actor is a process with a state, a actor will process messages that has been sent to its mailbox.
  • A actor can only communicate by sending messages
  • A actor can supervise and create new actors.

Messages

  • Every actor has a mailbox where it messages will be placed
  • Messages should always be immutable
  • Messages are fire and forget

Supervision

  • A actor can supervise other actors
  • When a supervised actor fails the supervisor is informed and can choose to:
    • Resume the actor
    • Restart the actor
    • Terminate the actor
    • Or escalate the failure

    This can be done with one-for-one where every supervised actor is handled seperatly or all-for-one where the decision is applied to all supervised actors.

Designing a actor based system

  • Much like any organization actors will naturally form a top down hierachy
    • To protect actors and isolate errors a actor can delegate critical or slow tasks to a children actor
  • Actors should not block, if you need to do a blocking operation, delegate that operatino to a child
